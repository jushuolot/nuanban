    第一部分 产品规格

    版本：V1 精简栈 · 2025
    技术基线：uni-app（微信 + H5）+ PocketBase + Docker Compose（见 [TECH_STACK_SIMPLE.md]）

    1. 产品定位与价值

    1.1 一句话

    暖伴勤工连接福利院/养老院老人、家属与高校勤工学生，提供可分类、可定价的陪护与生活服务；平台统一收款、按规则结算，单应用多角色，无监督端。

    1.2 解决的问题
    角色痛点平台价值
    老人找可靠陪护难、沟通成本高附近学生列表、SKU 明码标价、订单可追踪
    家属异地无法到场、支付分散绑定老人、代付/账单、外出审批
    学生勤工渠道少、信息不对称附近需求、排班签到、收入透明
    机构派单与档案分散PocketBase Admin 统一档案、派单、导出
    1.3 产品边界（明确不做）
    • 无 独立监督员 App / 督导后台
    • 无 按学校拆分的多租户 SaaS（学校仅为统计与合作维度）
    • 无 V1 强制微信支付实装（字段与流程预留，可 stub）

    2. 用户与角色

    2.1 角色矩阵
    角色入口核心诉求管理权限
    老人小程序 / H5找陪护、预约、看订单、确认完成、一键求助—
    家属小程序 / H5绑定老人、代付/充值、账单、外出审批—
    学生小程序 / H5附近需求、接单、排班签到、服务记录、收入—
    机构管理员PocketBase Admin本机构老人、派单、SKU、合作、导出org_admin
    平台管理员PocketBase Admin多机构、结算配置、全局导出platform_admin
    2.2 单 App、多角色
    • 同一微信/账号可拥有多个 user_roles 记录。
    • 登录后选择或切换 当前角色（activeRole）；界面与数据请求按角色隔离。
    • 切换入口：各模式「我的」→「切换身份」（仅 roles.length > 1 时显示）。
    2.3 注册与审核
    角色注册要点审核
    老人机构预建档码 / 手机号匹配通常自动或机构确认
    家属绑定码 / 扫码绑定老人绑定关系生效即可
    学生选学校、学号、学院等user_roles.status: pending → active
    ---

    3. 服务与商品（SKU）

    3.1 类目与 SKU
    • 服务大类 service_categories：如陪护、康复协助、陪伴聊天、外出陪同。
    • 可售项 service_items（SKU）：名称、单价、默认时长、单位、是否需外出审批、上下架状态。
    3.2 定价与选购
    • 下单时选择 SKU，价格以 SKU 为准。
    • 老人可自选；家属可为已绑定老人代选；机构可在后台维护 SKU 与机构专属价（V1 可用统一价，差价字段预留）。
    3.3 服务包（可选 V1 简化）
    • 家属端「服务包」可为多个 SKU 组合或预付费次数；V1 可用单 SKU 订单代替，结构预留。

    4. 订单与状态机

    4.1 订单来源
    来源触发方说明
    老人自下单老人选 SKU、选/匹配学生后生成
    家属代下单家属为已绑定老人下单，通常需支付
    机构派单管理员Admin 指定老人与学生
    学生申请/抢单学生待机构或规则确认后进入待服务
    4.2 主状态流
    草稿 → 待支付 → 待接单 → [外出待批] → 待服务 → 服务中 → 待确认 → 已完成

    分支状态：已取消、退款中

    状态说明典型操作方
    草稿未提交老人/家属
    待支付需家属或预付政策家属支付
    待接单等待学生接单或机构派单学生/管理员
    外出待批SKU 需外出时家属审批
    待服务已接单未开始学生签到开始
    服务中进行中学生写服务记录
    待确认待老人/家属确认老人确认完成
    已完成可进入结算系统/管理员
    4.3 支付与结算

    支付

    • 待支付订单：家属微信支付（V1 可 stub，保留 payment_status、paid_at）。
    • 平台收款：资金进入平台商户号（上线配置微信商户/服务商）。
    • 老人下单若绑定家属且政策要求预付 → 进入「待支付」；机构月结可在后台标记。

    结算

    • 订单完成后生成 settlements：学生劳务费、机构分成等。
    • V1：后台列表 + 导出对账；批量打款流程可二期。

    5. 匹配与位置

    5.1 老人找陪护
    • 输入/使用老人或机构位置（经纬度）。
    • 按服务类目筛选可接单 学生（user_roles.role=student 且 active）。
    • 列表按距离排序，展示脱敏信息（昵称、学校、评分字段预留）。
    5.2 学生找需求
    • 附近 待接单 订单或可申请服务的老人池。
    • 支持拒单/接单；接单后进入排班。
    5.3 机构派单
    • 管理员在 Admin 选择老人、学生、时段，写入 orders + schedules。
    5.4 位置技术
    • V1：Haversine 球面距离，默认半径约 5km（可配置）。
    • 不需 PostGIS；二期可加地图选点、轨迹。

    6. 排班、签到与外出

    6.1 排班 schedules

    关联：订单、学生、老人、计划开始/结束时间、状态（待服务/进行中/已完成）。

    6.2 签到
    • 学生到点 GPS 签到（V1）；二期可加扫码、围栏。
    • 签到后订单可进入「服务中」。
    6.3 外出服务 outdoor_approvals
    • SKU 标记「需外出」时，订单进入「外出待批」。
    • 家属（可选机构）审批通过后，方可开始服务。
    6.4 一键求助（老人）
    • 老人端 SOS：记录时间、位置，通知机构管理员（V1 可写记录 + Admin 列表；推送二期）。

    7. 学校与合作（非租户）

    7.1 原则
    • 全平台 一套数据，不按学校拆库、拆 Admin。
    • school_dict 仅为字典与统计维度。
    7.2 实体
    实体用途
    school_cooperation某校与某养老院/机构合作范围
    school_designated_elder某校项目指定的服务对象老人
    学生 school_id注册时选择，用于过滤合作范围内订单/老人
    ---

    8. 管理端（PocketBase Admin）

    8.1 功能清单
    • 老人档案 CRUD、关联机构/社区
    • 服务类目与 SKU 维护
    • 订单查询、改状态、派单
    • 院校合作与指定老人
    • 用户角色审核（学生 pending → active）
    • 数据导出：export_tasks（订单、结算、工时等）；V1 可同步导出小表
    8.2 明确不提供
    • 监督员工作台
    • 按校独立后台域名/数据隔离

    9. 客户端（uni-app）

    9.1 交付形态
    形态说明
    微信小程序主投放渠道
    H5浏览器、分享链接、非微信环境；与小程序共用业务代码
    9.2 分包结构
    分包主要页面
    pages/commonlaunch、login、role-select、register
    package-elder首页、找陪护、下单、订单列表/详情、设置
    package-family首页、绑定老人、待支付、账单、外出审批
    package-student首页、附近老人、接单、排班、签到、收入、资料
    深链统一经 pages/common/launch?role=&target=&id=，详见 [MINIAPP_ROUTING.md]。
    9.3 体验要求（老人端）
    • 大字号模式入口（设置页）
    • 关键操作步骤 ≤ 3 步完成下单（V1 目标）

    第二部分 技术架构

    目标：最低运维成本、最少组件、支持微信小程序 + H5，适合公益/试点项目快速上线。

    完整部署步骤见 [DEPLOY.md]。

    总览

    层级选型说明
    客户端uni-app（Vue 3）一套代码 → 微信小程序 + H5
    后端PocketBase单 Go 进程：SQLite、Auth、REST、Realtime、文件、内置 Admin
    部署Docker Composepocketbase 必选；caddy 可选（反代 + H5 静态）
    管理端PocketBase Admin (/_/)不维护独立 Vue Admin
    V1 不采用：NestJS + PostgreSQL + Element Plus 独立后台（运维与月租更高）。

    1. 客户端：uni-app Vue 3

    选型理由

    • 官方 mp-weixin + h5 双端，业务代码复用率高。
    • 仓库 packages/miniapp 已按三角色分包，仅需配置 API 与 H5 构建。

    常用命令

    cd packages/miniapp
cp .env.example .env
npm install
npm run dev:mp-weixin    # 微信开发者工具
npm run dev:h5           # 浏览器 H5
npm run build:h5         # 产出 dist/build/h5 → Caddy 或静态托管
npm run build:mp-weixin

    环境变量

    ##### PocketBase REST 根（不要 /v1）
VITE_API_BASE_URL=http://localhost:8090/api

    鉴权约定

    • Authorization: Bearer <pocketbase_token>
    • 多角色：user_roles 集合 + 请求头 X-Active-Role: elder|family|student
    • 复杂校验 → 二期 PocketBase Hooks

    2. 后端：PocketBase

    能力说明
    数据库内嵌 SQLite（pb_data/，备份=目录拷贝）
    API/api/collections/<name>/records
    认证内置 users 集合
    管理 UIhttp://host:8090/_/
    扩展pb_hooks/（Go/JS）支付、状态机、微信登录
    与 Supabase 自托管对比（简要）
    维度PocketBaseSupabase 自托管
    进程1 个二进制多容器（PG、Kong、Auth…）
    内存512MB～1GB 可跑通常 2GB+
    运维拷 pb_data 即备份PG 备份与组件升级
    适合本项目关系表 + CRUD + Admin强 RLS、大规模分析
    暖伴 V1 以 CRUD + 后台派单为主 → 选 PocketBase。

    3. 部署拓扑

      微信小程序 ──┐
  H5 浏览器 ──┼──► Caddy :443 ──► /api/* ──► PocketBase :8090
              │              └──► /      ──► H5 静态 (dist/build/h5)
              └──► 开发期可直连 :8090

    管理员浏览器 ──► PocketBase Admin /_/

    最小部署（仅 API + Admin）：

    docker compose up -d pocketbase

    完整部署（API + H5 同域）：

    npm run build:h5   # 在 packages/miniapp
docker compose --profile full up -d

    第四部分 部署指南

    本文档描述 V1 推荐栈：PocketBase +（可选）Caddy 反代 H5。约 15 分钟可在单机完成自托管。

    前置条件

    项要求
    服务器Linux（推荐 Ubuntu 22.04+），1 核 1G 起
    软件Docker 24+、Docker Compose v2
    域名（生产）已备案，解析到服务器（小程序需 HTTPS API）
    本地开发macOS / Windows + Docker Desktop 亦可

    目录与数据持久化

    nuanban/
├── docker-compose.yml
├── Caddyfile                    # full profile 时使用
└── packages/pocketbase/
    ├── pb_data/                 # SQLite + 上传文件（勿删）
    ├── pb_public/               # 可选静态资源
    └── pb_schema.json           # 集合导入

    pb_data 已在 .gitignore，备份 = 定期打包 pb_data 目录。

    一、仅启动 PocketBase（开发 / 最小生产）

    1. 克隆与进入目录
    git clone <your-repo-url> nuanban
cd nuanban
    2. 配置加密密钥（生产必做）
    export PB_ENCRYPTION_KEY="$(openssl rand -hex 16)"

    写入 .env 供 compose 读取（勿提交 git）
    
echo "PB_ENCRYPTION_KEY=${PB_ENCRYPTION_KEY}" > .env

    PB_ENCRYPTION_KEY 至少 32 字符；生产禁止使用 docker-compose.yml 中的默认值。

    3. 启动
    docker compose up -d pocketbase
docker compose ps
docker compose logs -f pocketbase
